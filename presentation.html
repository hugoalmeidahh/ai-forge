FORGE · ai-forge

Um método para desenvolver software
na era da AI

Padrões que humanos e agentes de AI seguem da mesma forma — arquitetura modular, standards em camadas e skills executáveis. Um único repositório, todas as stacks.

FORGE code review NestJS Node.js Golang Frontend
O problema

Cada dev — e cada AI — escreve
o mesmo sistema de um jeito diferente

✕ Sem método

Padrões vivem na cabeça das pessoas. Onboarding lento, review subjetivo, retrabalho constante. A AI gera código que "funciona", mas não parece com o resto do projeto.

✕ Docs que ninguém segue

Wikis desatualizadas, READMEs copiados e colados entre projetos que divergem em semanas. Não há como verificar aderência automaticamente.

✕ AI sem contexto

Agentes de AI são poderosos, mas sem padrões carregáveis eles improvisam: cada prompt gera uma arquitetura nova, cada sessão um estilo diferente.

✓ A oportunidade

Se os padrões forem arquivos versionados, seletivamente carregáveis e com precedência clara, a AI vira a executora mais disciplinada do time.

O insight

Padrão que a AI consegue seguir
é padrão que ela consegue carregar

O princípio central do método:

A AI só é coerente com os padrões se eles estiverem organizados
de forma que ela consiga carregá-los seletivamente,
na ordem certa, sem ambiguidade de precedência.
  • Cada standard é um arquivo .md pequeno e focado — não um documento gigante
  • Um index.md decide o que carregar por tipo de tarefa — economia de contexto
  • Distribuído como pacote (npm + pip) — versionado, atualizável, igual em todos os projetos

O padrão central

Módulos por domínio de negócio

Um módulo agrupa tudo de um contexto (user, purchase, notification). Regra de ouro: só o módulo dono manipula sua entidade — outros módulos chamam o service dele.

# fluxo canônico
Request
  → Controller   rota, auth, contrato — thin
  → Service      lógica de negócio, BD
  → Repository   persistência, providers
  → Service
  → Controller
  → DTO Output   contrato, oculta sensíveis
Response

Hierarquia

Standards em 4 camadas de precedência

L0 Company-wide — arquitetura modular, naming, git, segurança, observability, contrato de erro standards/ + core/
L2 Language — TypeScript strict, Go idiomático, JS moderno <stack>/standards/
L2 Stack — tradução do padrão modular para o framework (NestJS, Fastify, Go/Chi, React, Vue) <stack>/standards/
L3 Projeto — overrides pontuais e justificados. Sempre vence. CLAUDE.md / AGENTS.md

A AI carrega de cima para baixo; em conflito, a camada mais específica vence. Sem ambiguidade.

Code review · em detalhe

Review padronizado, não opinião de plantão

  • Parcimônia acima de tudo — sem ruído: se o comentário não muda a decisão de merge nem ensina algo, ele não existe
  • Anatomia fixa do comentário: o quê ajustar + por quê (regra do time citada) + impacto de não ajustar
  • Régua objetiva — a referência é o standard, não o gosto do revisor: "controllers-services: controller não acessa BD"
  • Análise estática incorporada — promise não-aguardada, catch vazio, SQL concatenado, segredo hardcoded, endpoint sem guard — sem depender de ferramenta externa
  • Tom colaborativo — sugerir, não impor; elogiar quando genuíno; 🔴 blockers são os únicos inegociáveis
🔴 🏗️ G — Controller acessando o banco direto

Mover a query para o service. Pelo padrão
controllers-services, o controller é fino —
quem lê/escreve no BD é o service.
Impacto: quebra a separação de camadas,
dificulta teste e foge do fluxo canônico.

// antes (no controller)
const user = await this.prisma.user.findUnique(...)
// depois
return UserOutput.getInstance(await this.userService.getById(id))

🔴 Blocker

Bug, segurança, camada quebrada, erro fora do BaseError, 5xx vazando detalhe. Muda antes do merge.

🟡 Sugestão

Melhora clara de qualidade ou aderência. Idealmente endereçada na própria PR.

🔵 Nitpick

Estilo e preferência. Opcional — condensado num bloco único, nunca espalhado.

Todo review fecha com um resumo global auto-suficiente: veredito (✅ / ⚠️ / 🔴), scorecard por pilar, bloqueadores e aderência aos standards — quem lê só ele entende o estado da PR.

Standards · em detalhe

Clean Code + DRY + 15 anos de estrada

Não é teoria de livro copiada — é curadoria: princípios clássicos filtrados pelo que funcionou (e quebrou) em produção ao longo de 15+ anos.

📖 Clean Code

Nomes revelam intenção — sem abreviação obscura. Função pequena, responsabilidade única: se o nome precisa de "and", faz duas coisas. Sem número mágico. Early return, aninhamento raso. Comentário explica o porquê, não o o quê.

♻️ DRY com critério

DRY é conhecimento duplicado — não linha parecida por acaso. Lógica repetida 2+ vezes → extrair para os helpers existentes antes de criar novo. Abstração prematura é dívida, não elegância.

🎯 Definições próprias

Regras nascidas da experiência: service por contexto (nunca por endpoint), módulo dono é o único que toca sua entidade, função nomeada em map/filter (stack trace melhor), if/switch explícito em vez de mapa dinâmico de funções.

Exemplos de regras opinativas

• Evitar try/catch quando o erro deve propagar — o filtro global resolve
• Escrita multi-tabela sempre em transação
• Sem parâmetro booleano que muda o comportamento da função
• Literal duplicado 3+ vezes → constante nomeada
• DTO é classe, nunca interface — validação viva, não só tipo

Por que isso importa

Cada regra existe porque um problema real a justificou. Elas são verificáveis — o FORGE review as cita pelo nome ao apontar violação. Padrão que não se verifica é só sugestão.
